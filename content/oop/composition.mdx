## Concept

**Composition** is a building block of Object-Oriented Design where complex objects are created by combining simpler, independent objects. Instead of an IS-A relationship (Inheritance), Composition forms a HAS-A relationship.

If an object is composed of other objects, creating the composite usually creates the parts, and destroying the composite destroys the parts (strong lifecycle dependency). A weaker form of this is termed *Aggregation*.

## When to use

- Favor composition over inheritance whenever possible! It provides greater flexibility.
- When you want to model a strict "whole/part" relationship.
- When the behavior of a component might change at runtime (you can swap composed objects, but you can't swap inherited classes).

## Gotchas

- **Deep Nesting**: Having too many layers of composition can make the system harder to trace.
- **Excessive Forwarding**: The composite class might become just a series of wrapper methods that blindly pass calls to the underlying components.
