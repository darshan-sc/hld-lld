## Concept

The **State** is a behavioral design pattern that allows an object to alter its behavior when its internal state changes. It appears as if the object changed its class.

Instead of writing massive `if (state == X) { ... } else if (state == Y) { ... }` blocks inside a context class, you extract all state-specific behaviors into individual State classes. The original object delegates all the work to its current State object.

## When to use

- When you have an object that behaves differently depending on its current state, the number of states is huge, and the state-specific code changes frequently.
- To avoid massive conditionals (Giant Switch Statements) that control how the object behaves based on a state variable.

interface State {
  insertCoin(): void;
  dispenseItem(): void;
}

// 2. The Context (Delegates behavior to current state)
class VendingMachine {
  private state: State;

  constructor() {
    this.state = new IdleState(this); // Initial state
  }

  public changeState(state: State): void {
    this.state = state;
  }

  // Delegated Actions
  public insertCoin(): void { this.state.insertCoin(); }
  public dispenseItem(): void { this.state.dispenseItem(); }
}

// 3. Concrete States
class IdleState implements State {
  constructor(private machine: VendingMachine) {}

  insertCoin(): void {
    console.log("Coin accepted. Please select an item.");
    // Transition to the next state!
    this.machine.changeState(new HasCoinState(this.machine));
  }

  dispenseItem(): void {
    console.log("Error: You must insert a coin first.");
  }
}

class HasCoinState implements State {
  constructor(private machine: VendingMachine) {}

  insertCoin(): void {
    console.log("Error: Coin already inserted. Select an item.");
  }

  dispenseItem(): void {
    console.log("Item dispensed! Enjoy.");
    // Transition back to idle
    this.machine.changeState(new IdleState(this.machine));
  }
}

## Gotchas

- **State vs Strategy**: They look identical (Context delegates to Interface). The difference is intent. In Strategy, the client determines which algorithm to use, and strategies usually aren't aware of each other. In State, the states *are* aware of each other and usually trigger transitions between themselves autonomously.
- **Tight Coupling**: Allowing States to transition themselves couples the State classes to each other. Alternately, the Context can handle the transitions (acting as a central State Machine orchestrator), but this often requires large `switch` statements, bringing you back to the problem you were trying to solve.
