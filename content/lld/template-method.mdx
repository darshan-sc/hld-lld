## Concept

The **Template Method** is a behavioral design pattern that defines the skeleton of an algorithm in the superclass but lets subclasses override specific steps of the algorithm without changing its overall structure.

Think of it like building a custom house. The construction company provides a standard template: build foundation, frame walls, add roof. But the *homeowner* decides the type of brick, the color of the walls, and the shingles on the roof.

## When to use

- When you want to let clients extend only particular steps of an algorithm, but not the whole algorithm or its structure.
- When you have several classes that contain almost identical algorithms with some minor differences. As a result, you might need to modify all classes when the algorithm changes.

## Gotchas

- **Breaking Liskov Substitution**: You might accidentally violate the Liskov Substitution Principle by suppressing a default step implementation via a subclass.
- **Rigid Skeleton**: Template methods tend to be harder to maintain the more steps they have.
- **Hollywood Principle**: "Don't call us, we'll call you." The superclass controls the workflow and calls the subclass methods, rather than the subclass calling superclass methods.
