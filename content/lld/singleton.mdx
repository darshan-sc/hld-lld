## Concept

Guarantee one instance with a shared access point. Keep the implementation minimal, but the boundaries explicit.

## When to use

- When you want a clear separation of concerns in object interactions.
- When interview discussions require trade-offs, not just syntax.
- When you need an example that maps quickly to production design decisions.

demo: Singleton = SingletonImpl()

## Gotchas

- Copying the pattern mechanically without checking the problem constraints.
- Ignoring object responsibilities, which usually leads to god classes.
