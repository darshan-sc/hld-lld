## Concept

The **Facade** is a structural design pattern that provides a simplified, higher-level interface to a complex subsystem of classes, libraries, or frameworks.

Instead of a client interacting with a dozen different deeply nested objects—knowing all their initialization orders and relationships—the client just calls one simple method on the Facade, and the Facade handles the ugly details.

## When to use

- When you need a limited but straightforward interface to a complex, messy subsystem.
- When you want to structure a subsystem into layers. You can create facades to define entry points to each layer.
- To decouple your core application logic from dense 3rd party frameworks (e.g., creating an `ImageConverterFacade` around a massive, messy C++ media manipulation library).

## Gotchas

- **God Object Transformation**: The Facade class can easily grow into a massive "God Object" coupled to every single class in your app if you're not careful. If it gets too big, consider extracting parts of it into additional facades.
- **Not a strict wrapper**: The Facade does *not* prevent client code from explicitly bypassing it and accessing the subsystem directly if they really want to. It's meant to be a convenient shortcut, not a security guard.
