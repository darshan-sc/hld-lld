## Concept

The **Proxy** is a structural design pattern that lets you provide a substitute or placeholder for another object. A proxy controls access to the original object, allowing you to perform something either before or after the request gets through to the original object.

Unlike a Facade (which simplifies a complex system) or an Adapter (which alters an interface), a Proxy provides the *exact same interface* as the underlying object.

## When to use

- **Virtual Proxy (Lazy Initialization)**: Delaying the creation of a heavy, memory-intensive object until it is absolutely needed.
- **Protection Proxy**: Checking access controls or permissions before allowing the client to call the underlying object.
- **Cache Proxy**: Caching the results of expensive requests (e.g., database queries or network downloads) to avoid hitting the actual object.

## Gotchas

- **Decorator vs Proxy**: They have identical structures (a class wrapping another class). However, their *intents* are different. A Decorator adds responsibilities dynamically. A Proxy controls access, handles lifecycle, or adds technical caching—it generally doesn't alter core business logic.
- **State synchronization**: If you are writing a Caching Proxy, be careful about cache invalidation! If the underlying data changes, the proxy will serve stale data unless you build an invalidation strategy.
