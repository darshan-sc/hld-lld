## Concept

The **Strategy** is a behavioral design pattern that lets you define a family of algorithms, put each of them into a separate class, and make their objects interchangeable.

It allows the algorithm (the strategy) to vary independently from the clients that use it. Wait, doesn't that sound like Polymorphism? Yes! Strategy is basically the textbook application of Polymorphism and Composition.

## When to use

- When you want to use different variants of an algorithm within an object and be able to switch from one algorithm to another during runtime.
- When you have a lot of similar classes that only differ in the way they execute some behavior. Just extract that behavior into a Strategy.
- To isolate the business logic of a class from the implementation details of algorithms that may not be that important in the context of that logic.

interface RouteStrategy {
  buildRoute(start: string, end: string): void;

  }

class WalkingStrategy implements RouteStrategy {

    console.log(\`Walking from \${start} to \${end} via Park Trails. ETA: 1h 10m\`);
  }
}

class TransitStrategy implements RouteStrategy {
  buildRoute(start: string, end: string): void {
    console.log(\`Taking Bus 42 from \${start} to \${end}. ETA: 45m\`);
  }
}

// 3. The Context (Maintains a reference to a Strategy)
class NavigationApp {
  private strategy: RouteStrategy;

  // The strategy can be injected or set later
  constructor(strategy: RouteStrategy) {
    this.strategy = strategy;
  }

  setStrategy(strategy: RouteStrategy): void {
    console.log("--- Switched Routing Strategy ---");
    this.strategy = strategy;
  }

  // The context delegates the heavy lifting to the strategy object
  executeRoute(start: string, end: string): void {
    this.strategy.buildRoute(start, end);
  }
}

// 4. Client Code
const app = new NavigationApp(new DrivingStrategy());
app.executeRoute("Home", "Office");

# Interface
class RouteStrategy(ABC):
    @abstractmethod
    def build_route(self, start: str, end: str) -> None: pass

# Concrete Strategies
class DrivingStrategy(RouteStrategy):
    def build_route(self, start: str, end: str) -> None:
        print(f"Driving from {start} to {end}")

class TransitStrategy(RouteStrategy):
    def build_route(self, start: str, end: str) -> None:

## Gotchas

- **Strategy vs State**: As mentioned in the State pattern, they share the same structure. However, in Strategy, the *client* dictates which strategy the context uses. In State, the states themselves cause the context to shift from one state to another.
- **Anonymous functions**: In modern languages with Functional Programming support (like JS/TS, Python, or Java 8+), you don't necessarily need to create distinct classes for Strategies. You can just pass an anonymous function (`lambda` or `arrow function`) representing the strategy logic directly into the context!
